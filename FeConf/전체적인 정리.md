# FeConf 2024 세션 정리

## 1. 웹 접근성 세션 (A11YKR)

### POUR 원칙

- **인지(Perceivable)** + **조작(Operable)** + **이해(Understandable, 예측가능성)** + **견고함(Robust)**

### WAI-ARIA를 통한 접근성 관리

- **역할(Role)** + **상태(State)** + **속성(Property)**

### 접근성 지원 Headless UI 라이브러리

- **Ark UI**
- **React Aria**
- **Melt UI**

### 접근성 트리

- 접근성 객체 모델(AOM)
- DOM 트리와 1:1 대응

### 접근성 테스트 방식

1. **Assertion 방식**
2. **Snapshot 방식**
   - `expect(getByRole('section')).toMatchAriaSnapshot()`

### 접근성 구현의 이점

- 사용자 편의성 향상
- SEO 개선
- 테스트 용이성
- **LLM으로의 확대 가능성**

### 학습 자료

- [웹접근성이 UI 설계에 중요한 이유](https://www.frontend.moe/posts/why-a11y-is-important-to-frontend-ui-design/)
- [A11YKR 커뮤니티](https://a11ykr.github.io/)

## 2. Query Router 전파 세션

### RSC (React Server Components)

- **서버에서만 알아야 하는 정보 활용**
  - 더 빠른 응답
  - 암호화 처리
  - 파일 처리

### TanStack과 RSC 결합

- `prefetchQuery` + `HydrationProvider` 활용
- RSC를 `useQuery`와 가까이 배치

### 기존 방식의 단점

- prefetch로 인한 복잡성 증가
- 직렬화 설계 실패 시 문제 발생 가능성

### 새로운 아키텍처 제안

#### 컴포넌트 구조

```
Component/
├── index.tsx
├── loader.tsx
└── client.tsx
```

#### 새로운 프레임워크: DataReady

- **쿼리 정의 특징**:

  - 캐시 가능한
  - 비동기
  - 읽기 연산
  - 같은 키 = 같은 쿼리

- `requiredResources` → `dataread().use()`로 대체

### 업계 사례

- **토스 김도한님**: 아래에서 위로 올려주는 resource 구조 제안
- **Isograph**: 강타입을 통해 아래에서 위로 올려주는 resource 구조 제안

## 3. React Native CodePush 대안 세션

### Hot Updater 구조

#### 플랫폼별 구현

- **iOS**: `bundleURL` 활용
- **Android**: `getJSBundleFile()` 활용

#### Hot Updater Plugin 구성요소

1. **Defined**: 번들 추출
2. **Storage**: 번들 저장 위치 관리
3. **Database**: 번들 정보 기록
4. **Console**: 코드푸시 제어 및 관리

### UUIDv7 활용

- **문자열 비교만으로 최신 UUID 식별 가능**
- 번들 업데이트 프로세스 단순화

### 품질 관리 아이디어

- 자유도가 있는 기능에 대해 **공통 테스트 케이스 제공**
- 테스트 통과 시에만 코드 반영 가능한 규칙 수립

### 깃헙코드
https://github.com/gronxb/hot-updater/tree/main

## 4. SEO 전략 세션

### 웹 퍼널 별 가치 전략

#### 검색 노출 단계

- **브랜딩 가치**: 유입 가능성 증가
- **주요 지표**: LCP, 크롤링 버짓

#### 검색 유입 단계

- **CAC(고객 획득 단가)** 감소: 비용/고객수 최적화
- **주요 요소**: 메타태그, 시맨틱 마크업

#### 전환 단계

- **LTV(고객 생애 가치)** 증가: 이탈율 감소, 전환율 증가
- **핵심**: 데이터 구조화

### 전략 1: 페이지 노출량 증대

#### 검색엔진 수집 단계

발견 안됨 → **발견** → **크롤링됨** → **인덱싱됨**

#### 발견성 개선

- `button` → `a` 태그 활용
- sitemap & robots.txt 제공
- CSP → SSR 전환

#### 크롤링 최적화

- **크롤링 버짓 관리**
  - 페이지 수집 시간 단축
  - **LCP 개선이 핵심**
- 의도적 noIndex 설정
- canonical 설정

### 전략 2: 클릭 유도 최적화

- **메타태그**: 지역별 언어 최적화
- **시맨틱 태그**: 다채롭고 정확한 데이터 노출
- **구조화된 데이터**: 검색/조회수 등 수치 노출
- 외부/내부 링크 관리
- 저자, 컨텐츠 관리

### 주요 측정 지표

- 총 클릭 수
- 총 노출 수
- 평균 CTR
- 평균 게재순위

## 5. Zero Cost CSS 세션

### DevUp-UI

- **GitHub**: https://github.com/dev-five-git/devup-ui
- **문서**: https://devup-ui.com/components/overview

### 기술적 접근

- **Rust + 웹어셈블리** 활용
- **코드 기반 중심** 접근법

### 기존 솔루션의 한계

- **Kuma UI**: 런타임 이슈 근본적 해결 불가

### 핵심 아이디어

- **React 코드 → HTML + CSS 클래스** 변환
- **동적 스타일링**: CSS 변수 활용
- **조건부 스타일** 지원

### 최적화 기법

- **CSS Tree Shaking**
- **27/38진법** 적용으로 빌드 결과 압축
